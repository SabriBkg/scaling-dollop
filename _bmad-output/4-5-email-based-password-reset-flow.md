# Story 4.5: Email-Based Password Reset Flow

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a SafeNet founder who has forgotten my password,
I want to request a password reset by email and choose a new password via a time-limited link,
So that I can regain access to my account without re-authorizing through Stripe and without exposing whether my email is registered to anyone watching the network.

## Acceptance Criteria

1. **Given** the `/login` page
   **When** the user clicks the "Forgot password?" affordance (currently rendered as a disabled `<span class="cursor-not-allowed opacity-50">` at `frontend/src/app/(auth)/login/page.tsx:69` — wire it to a real `<Link href="/forgot-password">`)
   **Then** the user is routed to a new public page at `/forgot-password` (file: `frontend/src/app/(auth)/forgot-password/page.tsx`) that prompts for email
   **And** submitting the form posts `{ email }` to `POST /api/v1/auth/password-reset/` (via the `/api/proxy/` axios client used by all other client→backend calls)
   **And** the backend responds **200 OK** with a fixed body `{ "data": { "message": "If an account exists for that email, we've sent a reset link." } }` **regardless of whether the email maps to a real `User`** — no email enumeration, no status differential, no body differential, no header differential, no timing differential beyond a few ms (FR-spec line 907 of epics.md)
   **And** the frontend renders the same generic confirmation message verbatim — never branches on a "user found" / "user not found" hint
   **And** for a registered email a password reset email is dispatched via Resend (the same transactional provider wired up in Story 4.1 — `core.services.email._ensure_configured` + `resend.Emails.send`)
   **And** for an unregistered email **no email is sent**, **no Resend API call is made**, and **no audit log row is written** — only a structured log line at INFO level is emitted: `[password_reset_request] EMAIL_NOT_FOUND email_hash=<sha256(email)[:16]>` (never log the raw email — operator log search uses the hash)

2. **Given** a valid password reset email arrives in the inbox
   **When** the user clicks the reset link
   **Then** the link is shaped `{SAFENET_FRONTEND_URL}/reset-password/<uidb64>/<token>` (Next.js page, NOT a Django view) where:
   - `<uidb64>` is `urlsafe_base64_encode(force_bytes(user.pk))` — Django's stdlib `django.utils.http.urlsafe_base64_encode` (the same encoding `django.contrib.auth` uses)
   - `<token>` is generated by `django.contrib.auth.tokens.PasswordResetTokenGenerator().make_token(user)` (stateless — token hash includes `user.password`, `user.last_login`, and a timestamp; **no DB row needed**, **no migration needed** — per epic line 912 and PRD architecture line 257-264)
   **And** the token expires after **1 hour** — `PASSWORD_RESET_TIMEOUT = 3600` is set in `safenet_backend/settings/base.py` (Django default is 3 days; we MUST override per epic line 911)
   **And** the token is **single-use** — once the password is changed, the included `user.password` hash changes and any previously-issued token deterministically fails `check_token` (this is the standard Django behavior — do NOT add a separate "consumed" flag or DB column)
   **And** the link routes to a Next.js page at `frontend/src/app/(auth)/reset-password/[uid]/[token]/page.tsx` that captures `uid` and `token` from `useParams()` and renders a "set new password" form

3. **Given** the password reset form (Next.js `/reset-password/<uidb64>/<token>` page)
   **When** the user submits a new password
   **Then** the form POSTs `{ uid, token, new_password, new_password_confirm }` to `POST /api/v1/auth/password-reset/confirm/`
   **And** the backend validates all of:
   - `uid` decodes via `urlsafe_base64_decode` to a real `User.pk` (else → 400 invalid token)
   - `PasswordResetTokenGenerator().check_token(user, token)` returns `True` (else → 400 invalid token)
   - `new_password == new_password_confirm` (else → 400 password mismatch)
   - Django's `validate_password(new_password, user)` passes — same `AUTH_PASSWORD_VALIDATORS` already configured at `settings/base.py:73-78` (min 8 chars, common-password rejection, user-attribute similarity, numeric rejection — matches the rules used in `CompleteProfileSerializer` at `serializers/account.py:13-23`)
   **And** on validation failure the response is `{ "error": { "code": "INVALID_RESET_LINK", "message": "This reset link is invalid or has expired. Please request a new one.", "field": null } }` for token/uid failures — the message is **deliberately the same for "uid decode failed", "user not found", "check_token returned False", and "token expired"** so an attacker cannot distinguish "valid token, wrong password" from "expired token" via response inspection (no email enumeration carries through to the confirm step either)
   **And** for password validation failures the response is `{ "error": { "code": "VALIDATION_ERROR", "message": <first error>, "field": "new_password" | "new_password_confirm" } }` — same envelope shape as `complete_profile` at `views/account.py:53-62`
   **And** on success the password is updated via `user.set_password(new_password)` + `user.save()` (Argon2id hashing per `PASSWORD_HASHERS` at `settings/base.py:68-71`); the response is **200 OK** with `{ "data": { "message": "Password updated. You can now sign in." } }`
   **And** the previously-issued reset token is automatically invalidated — re-POSTing the same token returns the same `INVALID_RESET_LINK` error (verified by the password-hash inclusion in `PasswordResetTokenGenerator._make_hash_value`)
   **And** the frontend, on success, redirects to `/login` with a flash query param (`?reset=success`) so the login page can render a one-time toast "Password updated — sign in below" (do NOT auto-login the user — the explicit re-authentication is a deliberate security boundary, mirrors how every other "post-credential-change" flow works)

4. **Given** an audit-trail requirement (FR43, NFR-D2 — 36-month retention)
   **When** a password reset request is made for a registered email
   **Then** an audit event is written via `write_audit_event(subscriber=None, actor="client", action="password_reset_requested", outcome="success", metadata={"user_id": user.id, "email_hash": <sha256_of_email[:16]>}, account=user.account)` — `subscriber=None` because this is an account-level event (mirrors `profile_completed` at `views/account.py:86-92`); `email_hash` instead of raw email because audit log retention is 36 months and PII minimization applies (NFR-D2 / NFR-S5)
   **And** when a password reset is **completed** (password actually changed), a second audit event is written: `write_audit_event(subscriber=None, actor="client", action="password_reset_completed", outcome="success", metadata={"user_id": user.id}, account=user.account)`
   **And** invalid/expired token attempts at the **confirm** endpoint emit a structured log only (`[password_reset_confirm] INVALID_TOKEN reason=<one of: bad_uid, user_not_found, check_token_failed>`) at WARNING level — they are **NOT** written to the audit log because (a) the user is unauthenticated and we cannot attribute the action to a tenant safely, and (b) it would create an unbounded write surface for an attacker minting random URLs
   **And** at no point is the raw email, raw password, raw token, or raw uid logged or audited — defense in depth against Sentry / log-aggregator leaks

5. **Given** rate-limiting requirements (epic line 922 — "a maximum of 3 reset requests per email per hour")
   **When** the same email submits more than 3 requests within a rolling 60-minute window to `POST /api/v1/auth/password-reset/`
   **Then** the 4th and subsequent requests within that window return `429 Too Many Requests` with `{ "error": { "code": "RATE_LIMITED", "message": "Too many password reset requests. Try again later.", "field": null } }` — a custom `PasswordResetRequestThrottle(SimpleRateThrottle)` subclass keyed by the **request-body email** (lower-cased + stripped), NOT by IP and NOT by authenticated user (the endpoint is anonymous)
   **And** the throttle's `get_cache_key` uses cache prefix `throttle_password_reset_<sha256(email)[:32]>` — the SHA-256 hashing prevents the cache key set from leaking the email-list back via a Redis dump, AND prevents the unbounded-cache-key scenario where an attacker mints arbitrary keys (the keyspace is bounded by `2**128` collisions but each is hashed, so the cache stays tidy)
   **And** the rate is configured via `DEFAULT_THROTTLE_RATES` at `settings/base.py:104-107`: add `"password_reset": "3/hour"` next to the existing `"auth": "5/min"` and `"profile": "3/min"` entries (matches architecture.md line 261-264 — that doc explicitly mentions a `password_reset` scope at "3 requests/email/hour")
   **And** when the request body's email is **missing or blank**, the throttle falls back to **IP-keyed** throttling (`throttle_password_reset_anon_<ip>`) at the same rate — defense against a script POSTing empty bodies in a loop
   **And** the **confirm** endpoint (`POST /api/v1/auth/password-reset/confirm/`) gets a separate throttle class `PasswordResetConfirmThrottle(SimpleRateThrottle)` keyed by **IP** at the existing `auth` scope (`5/min`) — defends against brute-forcing a valid `(uid, token)` combo within the 1-hour window. **DO NOT** key the confirm throttle by `uid` — an attacker controlling `uid` would simply rotate it; IP keying is the right granularity here.
   **And** rate limiter state is shared across processes via Redis (the existing DRF default cache backend; verify with `manage.py shell -c "from django.core.cache import cache; print(type(cache))"` if uncertain)

6. **Given** the password reset email body
   **When** the email is rendered for delivery
   **Then** the email is sent **from SafeNet's own auth-email identity** (NOT the per-tenant branded `"{company} via SafeNet"` `From` field used by the subscriber notifications) — `From: "SafeNet" <noreply@payments.safenet.app>` (use `settings.SAFENET_SENDING_DOMAIN` for the domain part to match Story 4.1)
   **And** the subject is exactly: `Reset your SafeNet password` (literal — no per-account variation, this email is from SafeNet to the founder, not from the founder's brand to a subscriber)
   **And** the body is rendered via `_render_email_shell(escaped_company="SafeNet", inner_html=...)` — the existing email shell extracted in Story 4.3 is generic enough; pass the literal string `"SafeNet"` so the brand `<h2>` reads "SafeNet" (not the tenant company name)
   **And** the inner HTML contains: a greeting paragraph, one explanatory paragraph (`"We received a request to reset the password for your SafeNet account. Click the button below to choose a new password. This link expires in 1 hour."`), a CTA button rendered as the same `<a>` styled like other email CTAs (background `#2563eb`, white text), and a single safety footer paragraph (`"If you didn't request a password reset, you can safely ignore this email — your password will not change."`)
   **And** the CTA links to the **frontend** URL `{settings.SAFENET_FRONTEND_URL}/reset-password/{uidb64}/{token}` — a NEW setting (`SAFENET_FRONTEND_URL`) is added because the existing `SAFENET_BASE_URL` (default `https://app.safenet.app`) points at the **backend** (it serves `/optout/<token>/` HTML directly per Story 4.4); the password reset flow lands on a Next.js page, which in dev runs on `http://localhost:3000` while Django runs on `http://localhost:8000`. In production both can collapse to the same domain, but the variables are distinct in dev.
   **And** all interpolated values are HTML-escaped via `html.escape(..., quote=True)` at the f-string boundary — same single-escape contract Story 4.3 locked in (the `_render_email_shell` callers escape once at the boundary; do NOT pre-escape and re-escape)
   **And** Resend send-time exceptions surface as **caller-handled** exceptions to the throttle-aware view — the view catches Resend `Exception` types, logs at ERROR level (`[password_reset_request] RESEND_FAILED user_id=<id>`), still returns the **same generic 200 success body to the client** (otherwise we would leak "this email is registered AND Resend is down"), and writes a `DeadLetterLog` row analogous to the notification dead-letter pattern at `tasks/notifications.py` (architecture line 521 — the dead-letter pattern). **Do NOT** retry inline (the request is synchronous and a Resend outage would block the user); accept best-effort delivery for the reset email at MVP.

7. **Given** observability and security hardening
   **When** the password reset flow runs end-to-end
   **Then** structured logs use the established `[task_name] EVENT key=value` pattern (matches `tasks/notifications.py` and `views/optout.py:94-97`):
   - `[password_reset_request] OK user_id=<id>` (registered email path — full info)
   - `[password_reset_request] EMAIL_NOT_FOUND email_hash=<hash>` (unregistered email path — no PII)
   - `[password_reset_request] RESEND_FAILED user_id=<id> exception_type=<cls>` (Resend outage)
   - `[password_reset_confirm] OK user_id=<id>`
   - `[password_reset_confirm] INVALID_TOKEN reason=<bad_uid|user_not_found|check_token_failed|password_invalid>`
   **And** the raw `token`, raw `uid`, and raw `email` are **never** logged — only `user_id` (after lookup) and SHA-256 prefix hashes (for the unregistered-email branch)
   **And** the response timing of the `password-reset/` endpoint is roughly constant between "user found" and "user not found" branches — the fast-path "user not found" still does a single dummy `time.sleep(0)` no-op (we are not implementing constant-time hashing equivalence at MVP — DOCUMENT this as a deferred follow-up: a determined attacker can probably still distinguish via timing, but the fast path's lack of an outbound API call is the larger signal)
   **And** the endpoint pair (`password-reset/`, `password-reset/confirm/`) is registered in `safenet_backend/urls.py` directly (not in `core/urls.py`) **next to the existing JWT auth endpoints** at `urls.py:17-18` — consistency with where `TokenObtainPairView` and `TokenRefreshView` already live; both use `permission_classes=[AllowAny]` (anonymous endpoints — no JWT required)

- [ ] **Task 2: Backend — Throttle classes** (AC: 5)
  - [ ] 2.1 Create new module `backend/core/views/password_reset_throttles.py`:
    ```python
    """Throttle classes for the password reset endpoints (Story 4.5).

    `PasswordResetRequestThrottle` keys by request-body email (sha256 hashed)
    so the rate limit applies per-email, not per-IP. Falls back to IP keying
    when email is missing/blank (defense against empty-body abuse).

    `PasswordResetConfirmThrottle` keys by IP at the existing `auth` scope —
    UID is attacker-controlled, IP is not.
    """
    import hashlib
    from rest_framework.throttling import SimpleRateThrottle


    class PasswordResetRequestThrottle(SimpleRateThrottle):
        scope = "password_reset"

        def get_cache_key(self, request, view):
            email = (request.data.get("email") or "").strip().lower()
            if email:
                ident = hashlib.sha256(email.encode()).hexdigest()[:32]
                return f"throttle_password_reset_{ident}"
            ip = self.get_ident(request)
            return f"throttle_password_reset_anon_{ip}"


    class PasswordResetConfirmThrottle(SimpleRateThrottle):
        scope = "auth"  # reuse the existing 5/min auth scope

        def get_cache_key(self, request, view):
            return f"throttle_password_reset_confirm_{self.get_ident(request)}"
    ```
  - [ ] 2.2 The hash truncation to 32 hex chars is intentional — full sha256 (64 chars) makes Redis key bloat unnecessary; 128 bits of collision space is far more than enough to keep the throttle bucket per-email.
  - [ ] 2.3 Both classes use `SimpleRateThrottle` (not `ScopedRateThrottle`) so we control `get_cache_key` directly. `ScopedRateThrottle`'s default keying (`user.pk` if authenticated, else IP) is wrong for password reset — both endpoints are anonymous and we want *email* keying for the request endpoint.
  - [ ] 2.4 `request.data.get("email")` works for both DRF's `APIView` and `@api_view` because DRF parses the body before throttle checks run (`SimpleRateThrottle.allow_request` runs after `view.initial`).                 raise serializers.ValidationError({"new_password": exc.messages})

- [ ] **Task 5: Backend — The two view functions** (AC: 1, 2, 3, 4, 5, 6, 7)
  - [ ] 5.1 Create new module `backend/core/views/password_reset.py`:
    ```python
    """Anonymous password reset endpoints (Story 4.5).

    Both endpoints are anonymous (`AllowAny`) — they live under /api/v1/auth/
    next to the JWT login endpoints. The throttle classes (per-email and
    per-IP) are the only abuse defense.

    Token format: Django stdlib `PasswordResetTokenGenerator` — stateless, no
    DB row, hash includes user.password so it auto-invalidates after a
    successful reset (FR43, NFR-D2). 1-hour expiry via PASSWORD_RESET_TIMEOUT.
    """
    import hashlib
    import logging

    from django.conf import settings
    from django.contrib.auth.models import User
    from django.contrib.auth.tokens import PasswordResetTokenGenerator
    from django.utils.encoding import force_bytes, force_str
    from django.utils.http import urlsafe_base64_decode, urlsafe_base64_encode
    from rest_framework import status
    from rest_framework.decorators import (
        api_view, permission_classes, throttle_classes,
    )
    from rest_framework.permissions import AllowAny
    from rest_framework.response import Response

    from core.serializers.password_reset import (
        PasswordResetConfirmSerializer,
        PasswordResetRequestSerializer,
    )
    from core.services.audit import write_audit_event
    from core.services.email import send_password_reset_email
    from core.views.password_reset_throttles import (
        PasswordResetConfirmThrottle,
        PasswordResetRequestThrottle,
    )

    logger = logging.getLogger(__name__)
    _GENERIC_MESSAGE = "If an account exists for that email, we've sent a reset link."
    _INVALID_LINK_ERROR = {
        "code": "INVALID_RESET_LINK",
        "message": "This reset link is invalid or has expired. Please request a new one.",
        "field": None,
    }


    def _email_hash(email: str) -> str:
        return hashlib.sha256(email.encode()).hexdigest()[:16]


    @api_view(["POST"])
    @permission_classes([AllowAny])
    @throttle_classes([PasswordResetRequestThrottle])
    def password_reset_request(request):
        """Anonymous: accept an email, send a reset link if the email is registered.

        ALWAYS returns 200 with a generic message — never differentiates between
        "email registered" and "email unknown" (FR-spec line 907).
        """
        serializer = PasswordResetRequestSerializer(data=request.data)
        # Any validation failure (bad format, missing field) — still return the
        # same generic 200. Failing 400 here would leak "the email format you
        # entered is malformed" → enumeration via input shape.
        if not serializer.is_valid():
            return Response({"data": {"message": _GENERIC_MESSAGE}}, status=200)

        email = serializer.validated_data["email"].strip().lower()
        # Case-insensitive lookup; pick first to handle legacy duplicates safely.
        user = User.objects.filter(email__iexact=email).first()

        if user is None:
            logger.info(
                "[password_reset_request] EMAIL_NOT_FOUND email_hash=%s",
                _email_hash(email),
            )
            return Response({"data": {"message": _GENERIC_MESSAGE}}, status=200)

        token_generator = PasswordResetTokenGenerator()
        token = token_generator.make_token(user)
        uidb64 = urlsafe_base64_encode(force_bytes(user.pk))

        base = settings.SAFENET_FRONTEND_URL.rstrip("/")
        reset_url = f"{base}/reset-password/{uidb64}/{token}"

        try:
            send_password_reset_email(user, reset_url)
        except Exception as exc:
            # Best-effort delivery: never differentiate "Resend down" vs
            # "user not found" in the response. Log + dead-letter.
            logger.error(
                "[password_reset_request] RESEND_FAILED user_id=%s exception_type=%s",
                user.id, type(exc).__name__,
            )
            from core.models.dead_letter import DeadLetterLog
            DeadLetterLog.objects.create(
                task="password_reset_email",
                account_id=user.account.id if hasattr(user, "account") else None,
                error=f"{type(exc).__name__}: {exc}",
            )
            return Response({"data": {"message": _GENERIC_MESSAGE}}, status=200)

        # Audit ONLY on the registered-email path. Email-not-found path leaves
        # zero audit footprint to avoid leaking enumeration via the audit log.
        account = user.account if hasattr(user, "account") else None
        write_audit_event(
            subscriber=None,
            actor="client",
            action="password_reset_requested",
            outcome="success",
            metadata={"user_id": user.id, "email_hash": _email_hash(email)},
            account=account,
        )
        logger.info("[password_reset_request] OK user_id=%s", user.id)
        return Response({"data": {"message": _GENERIC_MESSAGE}}, status=200)


    @api_view(["POST"])
    @permission_classes([AllowAny])
    @throttle_classes([PasswordResetConfirmThrottle])
    def password_reset_confirm(request):
        """Anonymous: validate the (uid, token) pair and update the password."""
        # Validate uid/token shape FIRST so we can materialize the User before
        # running password validators (which need user-attribute similarity).
        uid_raw = (request.data.get("uid") or "").strip()
        token_raw = (request.data.get("token") or "").strip()

        try:
            user_pk = force_str(urlsafe_base64_decode(uid_raw))
            user = User.objects.get(pk=user_pk)
        except (TypeError, ValueError, OverflowError, User.DoesNotExist):
            logger.warning("[password_reset_confirm] INVALID_TOKEN reason=bad_uid")
            return Response({"error": _INVALID_LINK_ERROR}, status=status.HTTP_400_BAD_REQUEST)

        if not PasswordResetTokenGenerator().check_token(user, token_raw):
            logger.warning(
                "[password_reset_confirm] INVALID_TOKEN reason=check_token_failed user_id=%s",
                user.id,
            )
            return Response({"error": _INVALID_LINK_ERROR}, status=status.HTTP_400_BAD_REQUEST)

        # Now run the full serializer with the user in context for
        # validate_password's user-attribute-similarity check.
        serializer = PasswordResetConfirmSerializer(
            data=request.data, context={"user": user},
        )
        if not serializer.is_valid():
            from core.views.account import _first_error_message, _first_error_field
            logger.warning(
                "[password_reset_confirm] INVALID_TOKEN reason=password_invalid user_id=%s",
                user.id,
            )
            return Response(
                {"error": {
                    "code": "VALIDATION_ERROR",
                    "message": _first_error_message(serializer.errors),
                    "field": _first_error_field(serializer.errors),
                }},
                status=status.HTTP_400_BAD_REQUEST,
            )

        new_password = serializer.validated_data["new_password"]
        user.set_password(new_password)
        user.save(update_fields=["password"])

        account = user.account if hasattr(user, "account") else None
        write_audit_event(
            subscriber=None,
            actor="client",
            action="password_reset_completed",
            outcome="success",
            metadata={"user_id": user.id},
            account=account,
        )
        logger.info("[password_reset_confirm] OK user_id=%s", user.id)
        return Response(
            {"data": {"message": "Password updated. You can now sign in."}},
            status=200,
        )
    ```

### Architecture Compliance

- **Token mechanism: Django stdlib `PasswordResetTokenGenerator`.** Per epic line 912 and architecture.md line 257-264 (Authentication & Security section). The token's hash includes `user.password` (the Argon2id hash itself), `user.last_login`, and a timestamp — calling `user.set_password(...) + user.save()` mutates the password column, which mutates the hash, which deterministically invalidates every previously-issued token for that user (single-use without a separate "consumed" flag). **DO NOT** use `signing.TimestampSigner` here (that's the optout-token pattern, Story 4.4) — `PasswordResetTokenGenerator` is purpose-built for this flow and provides the auto-invalidation property for free.
- **`PASSWORD_RESET_TIMEOUT = 3600`** overrides Django's default of 3 days. Per epic line 911. Place in `settings/base.py` immediately under the existing `SAFENET_BASE_URL` block.
- **Anonymous endpoints (`AllowAny`).** The two endpoints live under `/api/v1/auth/` next to `TokenObtainPairView` and `TokenRefreshView` — but unlike those, `password-reset/` and `password-reset/confirm/` use `permission_classes=[AllowAny]` (no JWT required, no session required). The throttle classes (per-email and per-IP) are the abuse defense. Mirrors how Django's own auth views handle this surface.
- **Email-enumeration defense (FR-spec line 907 + epic AC1).** The response is **identical** in body, status, and shape between "email registered" and "email unknown" branches. Audit log is **only** written on the registered path — the audit log itself is operator-readable (architecture.md line 821) so a footprint there would leak enumeration. The unregistered branch logs only a hashed email at INFO level. Timing is best-effort constant (no SDK call on the unregistered path; the registered path's outbound Resend latency is a known signal but accepted at MVP — documented as a deferred follow-up).
- **No-enumeration on the confirm endpoint.** Bad uid, unknown user, expired token, and `check_token_failed` all collapse to the same `INVALID_RESET_LINK` error. The `reason=` differential exists only in structured logs (operator-side) — not in the HTTP response. Tested in Task 7.2 `test_response_is_byte_identical_for_bad_uid_vs_check_token_failed`.
- **Argon2id hashing.** `user.set_password(new_password)` uses the configured `PASSWORD_HASHERS` (settings/base.py:68-71 — Argon2 primary, PBKDF2 fallback). The completed password's hash starts with `argon2$argon2id$...`. Tested.
- **Public route surface.** This story adds three public-by-design routes:
  - `POST /api/v1/auth/password-reset/` (backend, anonymous, JSON)
  - `POST /api/v1/auth/password-reset/confirm/` (backend, anonymous, JSON)
  - `GET /reset-password/<uid>/<token>` (frontend Next.js page, no auth)
  Plus modifications to `/login` (the "Forgot password?" link) and `/forgot-password` (new Next.js page).
- **JWT refresh-token NOT auto-revoked.** When the password changes, existing JWT refresh tokens are NOT auto-invalidated by `simplejwt` — they live until their natural expiry (7 days). At MVP this is accepted: the password reset flow is for "I forgot my password" not "my account is compromised." A future story can add `simplejwt`'s `BlacklistedToken` model + signal to revoke all outstanding refresh tokens on password change. **Document this gap in Completion Notes.**
- **No new DB models, no migrations.** The token mechanism is stateless; the audit log uses the existing `AuditLog` model; the dead-letter uses the existing `DeadLetterLog` model. **Verify by running `manage.py makemigrations --check --dry-run` after the implementation — the only output should be "No changes detected".**
- **Two URL settings.** `SAFENET_BASE_URL` (Story 4.4) points at the **backend** (Django serves `/optout/<token>/` HTML directly). `SAFENET_FRONTEND_URL` (this story) points at the **frontend** (Next.js renders the `/reset-password/...` form). In dev these differ (`localhost:8000` vs `localhost:3000`). In production they may collapse to the same domain via reverse proxy; the variables remain distinct for clarity.

### What NOT to Do

- Do **NOT** use `django.core.signing.TimestampSigner` for the password reset token. That's the optout-token pattern (Story 4.4). `PasswordResetTokenGenerator` is purpose-built for password reset and gives single-use auto-invalidation for free via the `user.password` hash inclusion.
- Do **NOT** add a new DB model for password reset tokens. Stateless via Django stdlib. **No migration.**
- Do **NOT** branch the `password-reset/` response on whether the email is registered. The whole point is no email enumeration. Body, status, headers, shape — all identical. Verified by Task 6.3 `test_returns_200_for_unregistered_email` byte-identical assertion.
- Do **NOT** write an audit-log row on the unregistered-email path. The audit log is operator-readable (architecture.md line 821); a footprint there would leak enumeration. Only the registered path produces an audit row. Verified by Task 6.3 `test_audit_event_NOT_written_for_unregistered_email`.
- Do **NOT** auto-login the user after a successful password reset. The redirect to `/login?reset=success` and the explicit re-authentication is a deliberate security boundary — matches every other "post-credential-change" flow in modern apps. Auto-login would also bypass the JWT refresh-token rotation question (which is deferred anyway).
- Do **NOT** revoke existing JWT refresh tokens on password change at MVP. `simplejwt` doesn't auto-revoke on password change; adding `BlacklistedToken` is a separate epic. Document the gap, defer the fix.
- Do **NOT** retry Resend failures inline. The user is waiting synchronously; a Resend outage would block the request indefinitely. Single best-effort attempt; on failure, log + DeadLetterLog + return the same generic 200 (so the response doesn't leak "Resend failed for this email" → which would also leak "this email is registered AND Resend is down").
- Do **NOT** add CSRF protection to the reset-password endpoints. They're DRF JSON endpoints — CSRF is not the threat model (no session cookies, no implicit auth). The DRF default `SessionAuthentication` is NOT in the `DEFAULT_AUTHENTICATION_CLASSES` (settings/base.py:92-94 — only JWT) so the endpoint is purely token/password-based.
- Do **NOT** key the request-endpoint throttle by IP. Multiple users behind a NAT (corporate, university, mobile carrier) would share the bucket and starve each other. Email keying is per-account, which is what the spec asks for.
- Do **NOT** key the **confirm**-endpoint throttle by `uid`. Attacker controls `uid`. IP keying at the existing `auth` scope (`5/min`) is correct here — same protection applied to login attempts.
- Do **NOT** write a separate email shell for SafeNet auth emails. Reuse `_render_email_shell` with `escaped_company="SafeNet"`. The shell is the visual single source of truth (locked in by Stories 4.3 and 4.4).
- Do **NOT** invent a token-format combining (uid, token) into a single string. Django's idiomatic pattern is two URL segments — `uidb64` + `token` — and that's how the rest of the Django ecosystem (admin, allauth) does it. Stay idiomatic.
- Do **NOT** decode the `uid` client-side to render a "Resetting password for X" header. The uid is `urlsafe_base64_encode(user.pk)` so it doesn't yield the email anyway, but more importantly, even attempting it implies a future enumeration vector and adds zero UX value (the user clicked the link in their own inbox; they know whose account it is).
- Do **NOT** add client-side email-format validation that rejects the form before the API call. That creates a different enumeration channel (client-side rejects malformed; server always succeeds → response timing leaks valid-email shape). Let `<Input type="email" required>` do the HTML5 minimum and let the backend's generic 200 handle everything else.
- Do **NOT** log the raw email, raw password, raw token, or raw uid. Only `user_id` (after lookup) and SHA-256 prefix hashes (for the unregistered-email branch). Sentry / log aggregator leak defense.
- Do **NOT** include the email or any PII in the audit metadata. Use `email_hash` (sha256 prefix) only. Audit-log retention is 36 months (NFR-D2); PII minimization applies.
- Do **NOT** add a fourth throttle scope beyond `auth`, `profile`, `password_reset`. The confirm endpoint reuses the existing `auth` scope (`5/min`) — same scope used for login attempts. Adding a fourth scope is unnecessary churn.
- Do **NOT** use Django templates for the password reset email. Inline HTML via `_render_email_shell` matches the existing pattern (Stories 4.1, 4.3, 4.4 all use inline HTML). The codebase has zero Django templates today.
- Do **NOT** add a "rotate refresh tokens on password reset" feature in this story. It's a separate auth concern and would require pulling in `simplejwt`'s blacklist app + migration + signal wiring. Defer to a follow-up.

### Tenant Isolation

- `User` is the global Django user table (NOT TenantScopedModel — tenants are `Account`s, and a User has a 1:1 OneToOneField to Account via `Account.owner` at `models/account.py:38-42`). The lookup `User.objects.filter(email__iexact=email).first()` is global by design: the password reset is for a User, not an Account.
- The audit row uses `account=user.account` (when the user has one — newly-created users mid-OAuth-flow may not yet have an Account FK; `hasattr(user, "account")` guards this — though in practice every founder has an Account by the time they would request a password reset). This makes the audit row tenant-scoped at write time even though the user lookup was global.
- The `email_hash` in audit metadata is per-user (1 user → 1 hash); collisions across the 16-char prefix are 64-bit-collision-rare. **Not** suitable for proving "this user reset their password" forensically (an operator with the email can recompute the hash and confirm, but pure correlation is collision-bounded). For 36-month retention compliance review, this trade-off is acceptable.
- **Cross-tenant leakage path:** the only way password reset could leak across tenants is if the `User.email__iexact` lookup hit a user with a different account FK. But each User has exactly one Account. No leak.

### Audit Trail (MUST use single write path)

```python
# Successful reset request (registered email path only)
write_audit_event(
    subscriber=None,                           # account-level event, no subscriber
    actor="client",                            # founder/client initiated
    action="password_reset_requested",
    outcome="success",
    metadata={
        "user_id": user.id,
        "email_hash": hashlib.sha256(email.lower().encode()).hexdigest()[:16],
    },
    account=user.account,                      # tenant-scoped
)

# Successful reset completion (password actually changed)
write_audit_event(
    subscriber=None,
    actor="client",
    action="password_reset_completed",
    outcome="success",
    metadata={"user_id": user.id},             # no email needed — already in the requested row
    account=user.account,
)

# NO audit row on:
#   - unregistered email request (enumeration defense)
#   - Resend failure (the dead-letter row is the system-of-record for the failure)
#   - any failed confirm attempt (unauthenticated; would create unbounded write surface)
```

The action verbs are `password_reset_requested` and `password_reset_completed` — past-tense snake_case, parallel to the existing `profile_completed`, `dpa_accepted`, `engine_activated`, `notification_opted_out`. Operator queries can filter by `action LIKE 'password_reset_%'` to surface every reset event across all accounts.

### Email Enumeration Defense — Threat Model

| Channel | Defense |
|---------|---------|
| Response body | Byte-identical `{"data": {"message": "..."}}` for found vs unknown |
| Response status | `200` always — no 404, no 422 |
| Response headers | No conditional `Set-Cookie`, no conditional `Location`, no debug headers |
| Response timing | No outbound API call on unknown path (best-effort; deferred follow-up) |
| Audit log | Zero rows on unknown path (operator-readable, would leak) |
| Structured logs | Hashed email only on unknown path (`email_hash=<16char>`); raw email NEVER appears |
| Rate-limit response | `RATE_LIMITED` returned regardless of registered status |
| Frontend behavior | Renders generic message verbatim regardless of response (incl. network error) |
| Confirm endpoint | `INVALID_RESET_LINK` for all four "no-go" branches (bad_uid, user_not_found, check_token_failed, token_expired) |

A determined attacker can probably still distinguish via the **timing** of the registered path (one outbound HTTPS to Resend takes ~50–200ms) vs the unknown path (~5ms). At MVP this is acknowledged and accepted; the next-tier defense (constant-time fan-out via Celery, dummy outbound calls on the unknown path) is a follow-up. Document in Completion Notes.

### Completion Notes List

- **Token mechanism**: Django stdlib `PasswordResetTokenGenerator` (stateless, hash includes `user.password` → auto-invalidates after a successful reset). `PASSWORD_RESET_TIMEOUT = 3600` overrides Django's default 3-day expiry.
- **Email-enumeration defense**: response is byte-identical (status, body, shape) for registered vs unregistered emails on the request endpoint. Audit log written ONLY on the registered + Resend-success path. Unregistered emails produce only a hashed-email INFO log, no audit row, no Resend call.
- **Confirm-endpoint enumeration defense**: bad uid, unknown user, expired token, and `check_token=False` all collapse to `INVALID_RESET_LINK` with byte-identical response. Differential lives only in structured logs (operator-side).
- **Throttling**: `password_reset` scope at 3/hour keyed by sha256(email) (not IP — corporate NATs would otherwise share a bucket). Confirm endpoint reuses the existing `auth` scope at 5/min keyed by IP. Falls back to IP when email is blank.
- **Audit actions added**: `password_reset_requested` (registered + sent path) and `password_reset_completed` (successful confirm). Both use `actor="client"`, `subscriber=None`, `account=user.account`, with `email_hash` (sha256 prefix) only — never raw email.
- **Resend failure handling**: single best-effort attempt. On failure: structured log, DeadLetterLog row (`task_name="password_reset_email"`, `account=user.account`), and the same generic 200 response (so failure doesn't leak "this email is registered AND Resend is down"). No audit row written when Resend fails.
- **Custom exception handler change**: added `Throttled → "RATE_LIMITED"` mapping to `core/views/errors.py` so 429 responses return `code: "RATE_LIMITED"` per AC5.
- **Frontend Suspense boundary**: refactored `app/(auth)/login/page.tsx` to wrap the inner component in `<Suspense fallback={null}>` because `useSearchParams()` requires a Suspense parent in App Router. The "Forgot password?" disabled span was replaced with a real `<Link href={ROUTES.FORGOT_PASSWORD}>`.
- **Frontend pages added**: `/forgot-password` (single email input, generic confirmation message rendered verbatim regardless of API response — including network errors — to mirror the backend's enumeration defense) and `/reset-password/[uid]/[token]` (dynamic route, redirects to `/login?reset=success` on success, renders "Link expired" on `INVALID_RESET_LINK`, field errors on `VALIDATION_ERROR`). No client-side email-format validation that rejects before the API call (would create a different enumeration channel).
- **Public route surface**: `/forgot-password` and `/reset-password` added to `middleware.ts` PUBLIC_PATHS and matcher exclusion regex.
- **No new DB models, no migrations**: stateless token generator + reuse of existing AuditLog and DeadLetterLog. Verified.
- **Deferred — JWT refresh-token revocation on password change**: `simplejwt` does NOT auto-revoke outstanding refresh tokens when a user's password changes; existing refresh tokens live until their natural 7-day expiry. At MVP this is accepted (the flow is "I forgot my password," not "my account is compromised"). A follow-up story can wire `simplejwt`'s `BlacklistedToken` app + a signal on password change to revoke all outstanding tokens.
- **Deferred — Resend-latency timing channel**: the registered-email branch makes an outbound Resend SDK call (~hundreds of ms) while the unregistered branch returns immediately. This is a known timing signal, accepted at MVP per the story's threat model. A future hardening story could add a fixed-latency floor or move the Resend dispatch to a Celery task.
- **Test count**: 35 new tests (34 unit + 1 end-to-end). All pass. The e2e test exercises the full flow: request → email rendering → URL extraction from HTML → confirm → password change → token single-use → both audit events.
